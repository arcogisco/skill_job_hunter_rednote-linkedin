# Feishu API Notice

这份说明只记录目前这套 Feishu Bitable 写入里，已经真实踩到过的坑。

## 1. 权限不够时会直接失败

常见表现：

- 能拿到 `tenant_access_token`
- 但补字段时 `403 Forbidden`
- 或只能读表，不能写记录

建议：

- 至少开通 Bitable 读写相关权限
- 如果希望脚本自动补表头，还要给表结构编辑权限

## 2. 现成表如果只有默认列，会报 `FieldNameNotFound`

飞书新建表时，通常只有默认列，比如 `文本`。

如果直接往这种表写：

- `公司名称`
- `岗位名称`
- `关键词`
- `城市`

这类字段，API 会直接报 `FieldNameNotFound`。

现在脚本会在有权限时自动补齐默认表头。

## 3. 新表常常会带几行默认空白占位行

这就是之前“前 5 行是空的，从第 6 行开始写”的根因。

现在脚本已经改成：

- 先识别飞书默认空白行
- 优先回填这些空白行
- 不够时再继续 append 新记录

## 4. Feishu API 偶发会出现 SSL EOF

这次真实联调里，`urllib` 对飞书接口偶发出现：

- `EOF occurred in violation of protocol`

尤其是：

- 读字段
- 建字段

现在脚本已经做了两层兜底：

- 先走普通请求重试
- 如果还是撞到这个 SSL EOF，再自动退回 `curl` 路径

这个兜底现在已经覆盖到：

- token 获取
- 读字段 / 读记录
- 建字段
- 普通 API 请求

## 5. 当前 Feishu Bitable 默认字段

现在这套写入默认只需要这些列：

- `公司名称`
- `岗位名称`
- `一句话概括`
- `关键词`
- `城市`
- `Refer`
- `邮箱`
- `小红书链接`
- `LinkedIn 链接`
- `来源`
- `搜索批次ID`

不再需要这 3 列：

- `去重键`
- `首次发现时间`
- `最近更新时间`

去重仍然保留，但只作为脚本内部逻辑，按精确链接判断，不要求表里有对应列。

## 6. 没有现成 base 时，当前状态是“可行但还不够稳”

已经确认的事实：

- 可以直接通过 Feishu API 新建一个全新的 Bitable base
- 飞书会立即返回这个新 base 的链接、`app_token`、默认 `table_id`
- 拿到这些值以后，继续往默认表写入这条路在能力上是可行的
- 新版脚本已经改成：优先复用这个默认表，并先清掉默认空行/空列，再补我们自己的字段
- 有些租户下，`GET /bitable/v1/apps` 会直接 `404`
- 这种情况下不能靠“先列出现有 base 再匹配”，要直接走 `create_app`

但目前还不够稳的地方是：

- “不给任何现成 `app_token/table_id`，一条命令从 0 自动建完再写入”偶尔会卡很久
- 这不是单一业务规则问题，主要还是 Feishu API 在 token、字段、schema 这些环节偶发抖动

当前建议：

- 生产使用优先走这两条已经验证更稳的路径：
  - 复用已有表
  - 在已有 base 下新建一张表再写入
- “完全没有现成 base”的场景，先视为实验态能力

## 7. 新建 base 时，飞书默认表不止一列

真实看到的默认情况通常不是只有 `文本` 一列，还可能带这些默认列：

- `单选`
- `日期`
- `附件`

所以“新建表先清默认空行和空列”这条规则不能只盯着 `文本`，
而是要把所有不在我们目标 schema 里的默认列都清掉。

现在脚本会：

- 尽量把默认主列复用成我们需要的字段
- 删除无关默认列
- 删除默认空白占位行

## 8. 最终一定要回 Feishu 链接

不管是：

- 复用已有表
- 在已有 base 下新建表
- 还是未来完全自动创建新 base

最终都应该返回 Feishu Bitable 链接，方便用户直接跳转。
